# Book Format Reference

This document defines the exact output format for Candlekeep-optimized books. Every book generated by Book Forge follows this structure.

## Complete Book Template

```markdown
# [Book Title]

**Domain:** [domain]
**Audience:** AI agents performing [task]
**Coverage:** This book covers [inclusions]. It does NOT cover [exclusions].
**Rule count:** [N] rules across [M] chapters
**Last updated:** [YYYY-MM-DD]

---

## Table of Contents

1. [Chapter 1: Precise Title — Subtopic A, Subtopic B, Subtopic C](#chapter-1)
2. [Chapter 2: Precise Title — Subtopic D, Subtopic E](#chapter-2)
...

---

# Chapter 1: [Precise Title — Subtopics Listed]

> **Scope:** This chapter covers [X, Y, Z]. For [related topic A], see Chapter M.

### Rule 1.1: [Imperative Action Statement]
**Impact:** CRITICAL | HIGH | MEDIUM | LOW

[1-3 sentence description of the rule and why it matters.]

**Do:**
\`\`\`
[correct example with brief annotation]
\`\`\`

**Don't:**
\`\`\`
[incorrect example]
\`\`\`
*Why wrong: [1 sentence explaining the failure mode]*

**Verify:** [How to check this rule is followed — a command, a check, or a condition]

**Common mistakes:**
- [Mistake 1 — brief description]
- [Mistake 2 — brief description]

---

### Rule 1.2: [Next Rule Title]
...

## Related
- For [topic], see Chapter N: [Title] (Rules N.1-N.3)

---

# Chapter 2: [Next Chapter]
...
```

## Rule Numbering

Rules use `chapter.rule` format: Rule 3.1 is the first rule in Chapter 3. When appending new rules to an existing chapter, continue the numbering sequence. When inserting between existing rules, use decimal sub-numbering (e.g., 3.1a or renumber the chapter).

## Impact Rating Guide

| Rating | Meaning | Examples |
|--------|---------|---------|
| CRITICAL | System failure, data loss, security vulnerability, or complete workflow breakdown | Missing error handling on webhooks, SQL injection, infinite loops |
| HIGH | Significant bugs, performance degradation, or reliability issues | Inefficient queries, missing retry logic, incorrect data types |
| MEDIUM | Suboptimal but functional — leads to maintenance burden or minor issues | Poor naming, missing logging, hardcoded values |
| LOW | Style, convenience, or minor optimization | Formatting preferences, optional performance tweaks |

When the domain does not naturally fit severity ratings (e.g., design patterns, theoretical concepts), omit the Impact line entirely. Do not force ratings where they add no value.

## Examples from Published Candlekeep Books

### Example: Security Rule (from "Web Application Security for AI Agents")

```markdown
### Rule 4.3: Validate and Sanitize All User Input Before Processing
**Impact:** CRITICAL

Accept user input only after validating type, length, format, and range. Sanitize by removing or encoding dangerous characters before passing to any downstream system (database, shell, HTML renderer).

**Do:**
\`\`\`python
from bleach import clean
user_input = clean(request.form['comment'], tags=['b', 'i', 'em'], strip=True)
if len(user_input) > MAX_COMMENT_LENGTH:
    raise ValidationError("Comment too long")
\`\`\`

**Don't:**
\`\`\`python
user_input = request.form['comment']
db.execute(f"INSERT INTO comments VALUES ('{user_input}')")
\`\`\`
*Why wrong: Raw user input interpolated into SQL — enables SQL injection and XSS.*

**Verify:** Grep codebase for string interpolation in database queries (`f"...{` patterns near `.execute()`). Run a SAST tool with SQL injection rules enabled.

**Common mistakes:**
- Validating on the client side only (server must re-validate)
- Sanitizing output but not input (defense in depth requires both)
- Using a blocklist instead of an allowlist for permitted characters
```

### Example: Workflow Rule (for an n8n book)

```markdown
### Rule 2.5: Set Retry on Error for All HTTP Request Nodes
**Impact:** HIGH

Configure retry behavior on every HTTP Request node to handle transient network failures. Set retries to 3 with exponential backoff. Without retry, a single timeout causes the entire workflow execution to fail.

**Do:**
\`\`\`json
{
  "parameters": {
    "url": "https://api.example.com/data",
    "options": {
      "retry": {
        "enabled": true,
        "maxRetries": 3,
        "retryInterval": 1000,
        "retryIntervalMultiplier": 2
      }
    }
  }
}
\`\`\`

**Don't:**
\`\`\`json
{
  "parameters": {
    "url": "https://api.example.com/data"
  }
}
\`\`\`
*Why wrong: No retry configuration — a single 503 or timeout kills the workflow.*

**Verify:** Check every HTTP Request node in the workflow JSON for the presence of `retry.enabled: true`.

**Common mistakes:**
- Setting maxRetries too high (>5) causing long execution times on permanent failures
- Using fixed interval instead of exponential backoff (hammers the failing service)
- Not setting a timeout alongside retry (retries hang indefinitely)
```

## Page Sizing Guidance

In Candlekeep, each Level 1 heading (`#`) creates a page boundary. Level 2+ headings (`##`, `###`) remain within their parent page. Each chapter uses one `#` heading, so **each chapter = one Candlekeep page**.

**Target: 5-10 rules per chapter (page).**

This balances two constraints:
- **Too few rules per page** (1-3) wastes the user's read rate limit — each `ck items read` call counts against their quota, and small pages mean more reads to cover the same content
- **Too many rules per page** (15+) bloats the agent's context window with rules that may not be relevant to the current question

If a topic naturally yields more than 10 rules, split it into two chapters with distinct scope declarations. For example, "Error Handling" with 14 rules becomes "Error Handling Foundations" (7 rules) and "Advanced Error Recovery" (7 rules).

## Writing Style Checklist

- [ ] Every sentence is imperative or declarative — no questions, no "imagine...", no anecdotes
- [ ] No filler words: "basically", "simply", "just", "actually", "in order to"
- [ ] Code examples are complete and runnable (not pseudo-code unless unavoidable)
- [ ] Each rule stands alone — reading only this rule gives enough context to act on it
- [ ] Cross-references include topic description + chapter + rule numbers
- [ ] TOC entries list specific subtopics, not vague category names
